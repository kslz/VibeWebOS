# VibeWebOS 项目设计文档 v0.1

## 1. 项目名称

**VibeWebOS**

## 2. 项目定位

VibeWebOS 是一个运行在桌面浏览器中的中文 Web 模拟操作系统。

它的目标不是实现真实操作系统，也不是实现真实应用平台，而是提供一个看起来像桌面操作系统的 Web 环境。用户可以在其中打开窗口、使用浏览器、搜索应用，并通过 LLM 动态生成看起来合理的应用界面和交互反馈。

系统整体风格偏认真、偏真实操作系统体验，不做明显恶搞 UI。

## 3. 项目目标

第一版目标：

1. 实现一个类 Windows 11 风格的中文 Web 桌面系统。
2. 提供桌面、任务栏、开始菜单、当前时间、桌面图标和多窗口管理。
3. 提供应用搜索功能，用户可以输入应用名或自然语言描述。
4. 系统调用 DeepSeek API 生成 2 到 3 个候选应用。
5. 用户选择候选应用后，系统生成一个 HTML/CSS 形式的伪应用窗口。
6. 生成的应用应具备基础本地逻辑能力。简单计算、计数、筛选、展开折叠、表单校验、列表增删改、Tab 切换等交互应优先在生成应用自己的沙箱内通过本地 JavaScript 完成。
7. 用户在生成应用中点击按钮、点击链接、提交表单时，默认不调用 LLM。只有被明确标记为需要 LLM 的动作，或生成应用主动发出 LLM 请求事件时，系统才捕获操作并调用 LLM 生成下一步界面。
8. 提供内置浏览器，浏览器可以模拟访问网站和模拟搜索。
9. 浏览器主页采用类似 Google 的极简搜索页，但系统界面中文化。
10. 提供设置页面，支持深色/浅色主题切换。
11. 提供关于系统页面，关于系统的介绍文案可通过配置文件修改。
12. 所有 App、窗口状态和交互历史只在当前页面会话中保留，不做持久化保存。

## 7. 系统核心模块

### 7.1 DesktopShell 桌面外壳

负责展示整个 Web 操作系统的基础界面。

包含：

* 桌面背景
* 桌面图标
* 任务栏
* 开始菜单
* 当前时间
* 窗口容器
* 全局主题

### 7.2 WindowManager 窗口管理器

负责管理所有窗口。

窗口需要支持：

* 打开
* 关闭
* 拖动
* 缩放
* 最小化
* 最大化
* 多窗口并存
* 点击窗口置顶
* 任务栏显示已打开窗口
* 从任务栏恢复窗口

窗口状态只存在于当前页面内存中。

刷新页面后：

* 所有窗口消失
* 所有生成 App 消失
* 所有浏览器页面状态消失
* 所有交互历史消失

关闭窗口后：

* 窗口状态丢失
* 应用内容丢失
* 窗口级 LLM 上下文立即销毁
* 不做恢复

### 7.2.1 窗口级 LLM 上下文

VibeWebOS 中每个打开的生成 App 窗口、浏览器窗口都维护独立的窗口级 LLM 上下文。

窗口级 LLM 上下文只存在于当前页面会话中，不做持久化保存。

窗口关闭后，对应上下文必须立即销毁。

不同窗口之间的上下文互不共享，避免多窗口内容互相污染。

每个窗口只保存以下上下文：

* 当前 HTML
* 当前摘要
* 当前表单临时值
* 最近若干次交互摘要

每个窗口不保存完整历史对话。

后续 LLM 请求必须只携带当前窗口自己的上下文，不允许携带其他窗口的 HTML、摘要、表单值或交互摘要。

### 7.3 Taskbar 任务栏

任务栏包含：

* 开始按钮
* 已打开窗口列表
* 当前时间
* 系统基础状态区域

任务栏中的窗口项可以：

* 显示窗口标题
* 点击后切换窗口显示/最小化状态
* 显示当前窗口是否处于等待状态

### 7.4 StartMenu 开始菜单

开始菜单提供基础入口：

* 浏览器
* 应用搜索
* 设置
* 关于系统

开始菜单不需要复杂磁贴、最近文件、推荐项目。

### 7.5 AppSearch 应用搜索

应用搜索是核心内置 App。

用户可以输入：

* 真实应用名
* 不存在的应用名
* 对应用的自然语言描述

例如：

* Photoshop
* Excel
* 一个像 Steam 的游戏库
* 复古 Windows 风格音乐播放器
* 看起来很专业的股票交易软件
* 神秘黑客终端
* 一个公司内部用的运维监控平台

搜索后，前端请求后端接口，后端调用 DeepSeek API。

LLM 返回 2 到 3 个候选应用。

候选应用要求：

* 名称和介绍偏真实软件风格
* 不一定是真实存在的软件
* 可以稍显风趣
* 不能一眼假
* 不主动恶搞
* 用户搜索不存在的软件时，也生成看起来合理的类似应用

每个候选项包含：

* 应用名称
* 应用简介
* 应用类型
* 界面风格描述

用户点击候选项后，系统打开新窗口，并调用 LLM 生成该应用的初始 HTML/CSS 界面。

### 7.6 GeneratedAppWindow 生成应用窗口

生成应用窗口用于展示 LLM 生成的 App。

生成内容要求：

* 以 HTML/CSS/JavaScript 为主
* 允许基础表单元素
* 允许较开放的本地 JavaScript，用于完成不需要 LLM 的即时交互
* 允许通过 HTTPS CDN 引入常见前端库，例如 ECharts、Chart.js、Three.js、D3、Day.js、Lodash 等
* 允许在生成 App 内部使用浏览器端状态、网络请求和第三方库来实现更复杂的玩具功能
* 简单业务逻辑应优先在本地完成，例如计算器加减、计数器、筛选、排序、表单校验、局部状态切换、图表渲染、小游戏逻辑
* 只有需要理解自然语言、生成新内容、复杂推理、重写页面结构或调用模拟智能能力时，才请求 LLM
* 不访问真实文件系统
* 不访问真实系统 API
* 不执行真实支付
* 不执行真实登录
* 不执行真实攻击或违法操作
* 不访问或操控父页面、桌面外壳、任务栏、开始菜单或其他窗口

生成应用目标：

> 视觉合理、常见交互和复杂前端效果可即时响应；只有语义性、生成式或需要 LLM 推理的功能才按需请求 LLM。

用户输入可以在当前窗口生命周期内保留，但不做持久化保存。

### 7.7 BrowserApp 内置浏览器

浏览器是内置 App。

浏览器首页采用类似 Google 的极简搜索页，但界面中文化。

浏览器支持：

* 地址栏输入
* 模拟搜索
* 查看模拟搜索结果
* 点击搜索结果跳转
* 点击链接跳转
* 表单提交
* 输入 `https://xxx.com`
* 输入自然语言查询

浏览器不支持：

* 多标签页
* 收藏夹
* 下载
* 上传文件
* 真实网页访问
* 复杂历史记录
* 用户登录态保存

浏览器内容完全由 LLM 模拟生成。

浏览器页面不提示“AI 生成”“模拟内容”等说明。

只有关于系统页面中说明本项目是模拟系统。

### 7.8 SettingsApp 设置

设置页面第一版只提供：

1. 深色/浅色主题切换
2. 关于系统入口

不提供：

* 壁纸切换
* 语言切换
* API Key 前端配置
* 模型切换
* 复杂个性化设置

### 7.9 AboutApp 关于系统

关于系统页面展示 VibeWebOS 的说明。

关于系统介绍文案需要放在配置文件中，方便后续修改。

运行中的生成页面、生成 App、浏览器页面、搜索结果页面不显示“AI 幻觉”或“模拟内容”提示。

## 8. 用户交互触发规则

以下行为会触发 LLM 请求：

* 应用搜索框回车或点击搜索
* 用户选择候选应用
* 点击生成 App 中明确标记为需要 LLM 的按钮，例如带有 `data-vibe-llm-action` 的控件
* 点击生成 App 中明确标记为需要 LLM 的链接，例如带有 `data-vibe-llm-action` 的链接
* 提交生成 App 中明确标记为需要 LLM 的表单，例如带有 `data-vibe-llm-submit` 的表单
* 生成 App 内部 JavaScript 主动向父窗口发送 `vibewebos:llm-request` 消息
* 浏览器地址栏回车
* 浏览器搜索框回车
* 点击浏览器页面中的链接
* 提交浏览器页面中的表单

以下行为不触发 LLM 请求：

* 普通输入过程
* 生成 App 中未标记为 LLM 动作的普通按钮点击
* 生成 App 中未标记为 LLM 动作的普通链接点击
* 生成 App 中未标记为 LLM 动作的普通表单提交
* 生成 App 内部本地 JavaScript 处理的计算、计数、筛选、排序、展开折叠、Tab 切换、表单校验、列表增删改、图表渲染、小游戏逻辑、CDN 库驱动交互
* 鼠标悬停
* 窗口拖动
* 窗口缩放
* 窗口最小化
* 窗口最大化
* 窗口关闭
* 点击任务栏
* 打开开始菜单
* 切换深色/浅色主题

## 9. 等待状态设计

所有调用 LLM 的操作都需要显示等待状态。

等待状态规则：

1. 等待状态只锁定当前窗口，不锁定整个桌面。
2. 多个窗口可以同时处于等待状态。
3. 用户仍然可以操作其他窗口、任务栏、开始菜单和设置页。
4. 每次等待时随机显示一条中文等待文案。
5. 如果等待超过 5 秒，则随机切换到另一条文案。
6. 文案切换需要有轻微动画。
7. 所有等待场景使用同一组文案，不区分搜索、生成 App、浏览器跳转或交互响应。

等待文案示例：

* 正在协调算力
* 正在大炼钢铁
* 正在从沙子中炼硅
* 正在开除测试工程师
* 正在让显卡深呼吸
* 正在给电子加班费
* 正在把需求塞进上下文窗口
* 正在重启不存在的服务器
* 正在优化不存在的代码
* 正在把咖啡转换成界面
* 正在让窗口假装自己很忙
* 正在给按钮安排命运
* 正在检查今日黄历
* 正在召唤产品经理
* 正在把 CSS 熬成汤
* 正在尝试理解用户真正想要什么

## 10. 生成 HTML 的边界

LLM 可以为生成应用窗口生成 HTML/CSS/JavaScript。由于 VibeWebOS 是玩具类项目，生成 App 运行时采用开放优先策略：尽量少限制临时应用能力，让它可以使用 CDN 库、浏览器端状态和较复杂的前端逻辑。

开放运行时的核心边界不是“禁止大部分 Web API”，而是“生成 App 不能逃出当前窗口去操控 VibeWebOS 父页面或其他窗口”。

允许：

* HTML
* CSS
* 内联 JavaScript
* HTTPS 外部 JavaScript
* HTTPS CDN 库
* 普通事件处理属性
* 浏览器端状态，例如 `localStorage`、`sessionStorage`、`indexedDB`
* 浏览器端网络请求，例如 `fetch`、`XMLHttpRequest`、`WebSocket`
* 基础表单元素
* 按钮
* 链接
* 表格
* 卡片
* 菜单
* 输入框
* 文本域
* 选择框
* 复选框
* 图片
* SVG
* 图标
* Canvas
* WebGL
* 音视频元素
* iframe 内部弹窗或下载能力，前提是不破坏父窗口隔离

不允许：

* 未受沙箱隔离的自由 JavaScript
* 读取本地文件
* 影响桌面外壳
* 影响任务栏
* 影响开始菜单
* 影响其他窗口
* 访问父窗口 DOM
* 访问或导航顶层窗口
* 通过 `window.parent`、`window.top`、`opener` 操作 VibeWebOS
* 使用 `javascript:`、`vbscript:`、`data:text/html` 等危险协议
* 执行真实支付、真实登录、真实攻击、隐私窃取或违法操作

LLM 生成的 HTML/JS 只能作为窗口内容渲染，并且必须运行在 iframe sandbox 内。

生成应用 iframe sandbox 允许 `allow-scripts`、`allow-forms` 和 `allow-same-origin`，以兼容更多 CDN 库、浏览器状态和复杂前端能力。

生成应用 iframe 不允许 `allow-top-navigation`。因此生成应用可以操作自己的 DOM、加载 HTTPS CDN 资源、运行复杂前端逻辑和维护 iframe 内状态，但不能访问父页面 DOM、桌面外壳、任务栏、开始菜单、其他窗口或跨窗口上下文。

生成应用如需请求 LLM，必须通过系统定义的消息协议向父窗口发送请求。父窗口只接受来自当前 iframe 的消息，并只使用当前窗口自己的 LLM 上下文构造请求。

推荐 LLM 动作声明方式：

```html
<button data-vibe-llm-action="summarize">总结当前内容</button>
<form data-vibe-llm-submit="plan-next-step">...</form>
```

普通按钮、普通链接、普通表单默认由生成应用本地 JavaScript 处理，不自动触发 LLM。

* 登录框

* 支付页面
* 收银台
* 订单页面

* 影视化安全工具

但系统不允许提供真实能力：

* 不真实支付

## 12. 敏感输入处理

用户在生成页面中可能会输入各种内容。

为了避免误收集敏感信息，需要在前端或后端对敏感字段做脱敏处理。

建议规则：

| 输入类型   | 处理方式               |
| ------ | ------------------ |
| 普通搜索词  | 原文传给 LLM           |
| 普通文本框  | 原文传给 LLM           |
| 地址栏    | 原文传给 LLM           |
| 普通表单内容 | 原文传给 LLM           |
| 密码框    | 不传原文，只传“用户输入了密码”   |
| 支付密码   | 不传原文，只传“用户输入了支付密码” |
| 验证码    | 不传原文，只传“用户输入了验证码”  |
| 银行卡号   | 不传原文或只传脱敏摘要        |
| 身份证号   | 不传原文或只传脱敏摘要        |
| 手机号    | 不传原文或只传脱敏摘要        |
| 邮箱     | 可传，也可按配置脱敏         |
| 支付金额   | 可以传金额              |

LLM 只需要知道用户完成了什么动作，不需要获得真实敏感内容。

需要注意，生成 App 运行在玩具化开放运行时中，仍可能通过自身 JavaScript 发起网络请求。这里的脱敏只保护 VibeWebOS 父窗口转发给 LLM 的表单 payload，不等同于对生成 App 自身网络行为的完整数据防泄漏。

## 13. 图片和图标需求

生成 App 或浏览器页面需要图片时，可以采用以下来源：

1. 互联网相关图片
2. 现成 icon 库
3. 本地内置图标
4. SVG 矢量占位图

图片策略：

* 优先使用可访问的相关图片或图标资源
* 如果获取失败，则使用 SVG/矢量图绘制占位内容
* 图片只用于增强视觉表现
* 不要求图片内容真实准确
* 不要求完全复刻真实网站图片
* 可以使用近似品牌风格，但不要求精确还原

LLM 不直接决定最终图片资源是否可用。
LLM 可以描述需要什么图片，系统负责寻找或生成图片资源。

## 14. 状态管理

第一版所有状态只保存在前端内存中。

需要保存的运行时状态包括：

* 打开的窗口列表
* 每个窗口的位置
* 每个窗口的尺寸
* 每个窗口的层级
* 每个窗口是否最小化
* 每个窗口是否最大化
* 每个窗口标题
* 当前窗口 HTML 内容
* 当前窗口表单临时值
* 当前窗口最近交互摘要
* 每个生成 App 窗口、浏览器窗口的独立 LLM 上下文
* 当前主题模式

不持久化保存：

* 生成 App
* 浏览器页面
* 应用交互历史
* LLM 上下文
* 完整历史对话

窗口级 LLM 上下文管理规则：

* 只在当前页面会话内存在
* 刷新页面后全部销毁
* 窗口关闭后立即销毁对应上下文
* 不同窗口之间上下文互不共享
* 每个窗口只保存当前 HTML、当前摘要、当前表单临时值和最近若干次交互摘要
* 不保存完整历史对话
* 任意 LLM 请求只能读取并更新发起请求的窗口上下文
* 用户输入内容
* 窗口位置
* 主题设置
* 搜索记录

刷新页面后全部恢复默认状态。

## 17. LLM 输出要求

LLM 返回内容必须是 JSON，不直接返回裸 HTML。

其中 HTML 放在 JSON 字段中。

应用生成类响应结构：

```json
{
  "windowTitle": "string",
  "html": "string",
  "summary": "string"
}
```

应用搜索响应结构：

```json
{
  "results": [
    {
      "name": "string",
      "description": "string",
      "appType": "string",
      "styleHint": "string"
    }
  ]
}
```

浏览器响应结构：

```json
{
  "pageTitle": "string",
  "url": "string",
  "html": "string",
  "summary": "string"
}
```

后端必须校验：

* 是否是合法 JSON
* 是否包含必要字段
* HTML 长度是否超限
* 是否包含会逃出当前窗口隔离边界的标签、属性或脚本
* 是否包含访问父窗口、顶层窗口、其他窗口或桌面外壳的代码
* 是否包含危险协议
* 浏览器 App 页面是否继续保持模拟网页边界，不执行真实网页 JavaScript

如果校验失败，后端返回错误，前端显示降级界面。

## 18. Prompt 需求

后端需要维护 Prompt 模板。

至少包含：

1. 应用搜索 Prompt
2. 应用生成 Prompt
3. 应用交互 Prompt
4. 浏览器首页/搜索 Prompt
5. 浏览器页面交互 Prompt

Prompt 基础要求：

* 系统界面使用中文
* 生成内容偏真实软件风格
* 不要明显恶搞
* 可以稍显风趣
* 不要在生成页面中提示“AI 生成”“模拟内容”
* 只返回 JSON
* HTML 只能用于窗口内部
* 生成应用可以生成较开放的本地 JavaScript，用于不需要 LLM 的即时交互
* 生成应用可以通过 HTTPS CDN 引入常见前端库来实现图表、可视化、小游戏和复杂控件
* 普通交互默认本地处理，不要把所有点击都设计成 LLM 请求
* 只有明确需要 LLM 的控件才添加 `data-vibe-llm-action` 或 `data-vibe-llm-submit`
* 生成应用不得访问父窗口、顶层窗口、桌面外壳、任务栏、开始菜单或其他窗口
* 不生成真实危险能力
* 不声明自己访问了真实网站
* 生成视觉上合理的页面和交互反馈

## 19. 错误处理

需要处理以下情况：

* DeepSeek API 请求失败
* DeepSeek API 超时
* LLM 返回非 JSON
* LLM 返回字段缺失
* LLM 返回 HTML 不合规
* 用户输入为空
* 用户重复点击导致并发请求
* 当前窗口已经关闭但请求返回
* 网络异常
* 后端异常

错误显示规则：

* 错误只显示在对应窗口内
* 不影响整个桌面
* 提供重试按钮
* 保留当前窗口
* 如果有上一版可用 HTML，则保留上一版内容

## 23. 验收标准

第一版完成后，应满足以下验收标准：

1. 打开页面后能看到类 Windows 11 中文桌面。
2. 桌面有浏览器、应用搜索、设置、关于系统四个图标。
3. 可以通过桌面图标或开始菜单打开内置 App。
4. 多个窗口可以同时打开。
5. 窗口可以拖动、缩放、最小化、最大化、关闭。
6. 任务栏能显示当前打开窗口。
7. 应用搜索可以输入应用名或自然语言描述。
8. 搜索后返回 2 到 3 个候选应用。
9. 点击候选应用后能生成一个看起来合理的应用窗口。
10. 点击生成应用中的按钮或提交表单后，窗口内容会继续变化。
11. 浏览器首页是中文化的极简搜索页。
12. 浏览器可以输入 URL 或自然语言。
13. 浏览器可以生成模拟搜索结果。
14. 点击浏览器页面中的链接后可以生成下一页。
15. 等待 LLM 时只锁定当前窗口。
16. 多个窗口可以同时等待。
17. 等待文案随机出现，超过 5 秒会切换。
18. 设置页可以切换深色/浅色主题。
19. 关于系统文案来自配置文件。
20. 刷新页面后窗口和生成内容全部丢失。
21. 关闭窗口后状态丢失。
22. 页面中除关于系统外，不主动提示“AI 生成”“模拟内容”等说明。
23. 生成应用 JavaScript 只在 sandbox iframe 内运行；允许开放前端能力和 CDN 库，但不得访问父窗口、桌面外壳或其他窗口。
24. 不提供真实登录、支付、攻击、隐私窃取等能力。
25. 每个生成 App 窗口、浏览器窗口都有独立的窗口级 LLM 上下文。
26. 不同窗口的上下文互不共享，关闭窗口后对应上下文立即销毁。
27. DeepSeek API Key 不出现在前端代码中。
